# 《一骑红尘：荔枝争运战》策略设计文档

> 版本：v1.0  
> 用途：指导客户端决策引擎分阶段实现与调优  
> 依据：参赛选手任务书、通信协议  
> 当前基线：`l1-finish` 仅 S14 验核 → S15 交付；`litchi-client` 含基础寻路/资源/任务框架

---

## 1. 胜负与得分总纲

### 1.1 局内胜负

| 情况 | 结果 |
|------|------|
| 单方退赛 | 退赛方负 |
| 双方同帧退赛 | 无效局 |
| 均未退赛 | 比较最终总分；同分则局内平局 |

### 1.2 得分结构（实现优先级）

最终总分 = 皇榜任务分 + 送达基础分 + 用时分 + 好果数量分 + 鲜度品质分 + 破关悬赏分 − 惩罚分

**未交付惩罚（极高）：**

- 送达基础分、好果分、鲜度分、用时分 → **全部为 0**
- 皇榜任务分封顶 **80**（无里程碑）
- 破关悬赏分封顶 **25**（无完成额外 +20）

**第一战略阈值：普通皇榜任务基础分累计 ≥ 90**

| 累计任务基础分 | 影响 |
|----------------|------|
| < 90 | 送达基础分折减；用时分 × min(累计/90, 1) |
| ≥ 90 | 送达基础分满额 240；用时分不再折减 |
| 60 / 90 / 110 | 里程碑额外 +15 / +20 / +15（合计最多 +50） |

**分项上限速查：**

| 分项 | 上限 | 关键公式（已交付） |
|------|------|-------------------|
| 皇榜任务分 | 180 | min(180, 任务基础分累计 + 里程碑) |
| 送达基础分 | 240 | min(240, 120 + floor(任务基础分×4/3)) |
| 用时分 | 69 | floor((600−交付帧)/600×70) × 任务系数 |
| 好果数量分 | 180 | floor(剩余好果/100×180) |
| 鲜度品质分 | 180 | floor(剩余鲜度/100×180) |
| 破关悬赏分 | 100 | min(原始,80)+20（有悬赏时） |

### 1.3 策略总目标（按优先级）

```text
P0  稳定在线、每帧回包、零/极少非法动作（避免退赛与扣分）
P1  必须完成交付（好果>0、鲜度>0、已验核、在 S15）
P2  普通皇榜任务基础分 ≥ 90（建议 3×30 分任务）
P3  尽量早交付（用时分最高 69）
P4  交付时保住好果与鲜度（各最高 180）
P5  适度对抗：设卡/破关悬赏/窗口，不以牺牲 P1–P4 为代价
```

## 2. 时间与阶段规划（600 帧）

### 2.1 全局时间轴

| 阶段 | 帧范围 | 主目标 |
|------|--------|--------|
| 开局抢节奏 | 1–80 | 定路线、领首资源、做第 1 个 30 分任务 |
| 中段积累 | 80–320 | 冲 90 任务分、保鲜度、处理固定站点 |
| 天气应对 | 80–120 / 200–240 / 320–360 / 440–480 | 预告后 30 帧调整路线（每次持续 60 帧） |
| 后段入关 | 320–390 | 清障、S11/S13 交接、备宫门 |
| 宫宴冲刺 | ≥390（或 450 强制） | S14 验核 → S15 交付、终局急策 |
| 终局 | 交付后 | 仅 WAIT / 空心跳，禁止其他主动动作 |

### 2.2 宫宴冲刺（RUSH）触发条件（任一满足）

- 第 390–449 帧：任一主车队到达 S14
- 第 390–449 帧：任一主车队不在 S11/S12/S13，且到 S14 最短路线距离 ≤ 15
- 第 390–449 帧：任一主车队不在 S11/S12/S13，且基础最快路线 60 帧内可到 S15
- 第 450 帧：强制触发

**冲刺后变化：**

- 开放 S14 `VERIFY_GATE`（普通阶段提交会被拒绝）
- 开放终局急策（每队整局 1 次）
- **禁止新派小分队**（在途仍落地）

### 2.3 建议帧预算（基础地图参考）

| 项目 | 典型帧数 |
|------|----------|
| 单段固定处理（含窗口风险） | 4–7 + 争夺 |
| 资源领取 | 2 |
| T01/T06 任务 | 3 |
| T04 清障任务 | 6 |
| S14 宫门验核 | 6（破关令可 −3，最低 3） |

## 3. 路线与移动策略

### 3.1 路线选择原则

- 以 `start.edges[]` / `inquire.edges[]` 建图，**禁止写死地图**
- 合法相邻节点以当前公开状态为准（单向边不可逆行）
- 三条宏观路线：
  - **官道**：处理点相对少，任务点多，适合冲任务与速度
  - **水路**：鲜度损耗低（0.045/帧），登船 7 帧 + 暴雨减速严重
  - **山路**：鲜度损耗高（0.07/帧），障碍与山雾多，适合绕敌不为主力

### 3.2 移动成本模型（决策用）

```text
到站所需移动量 = ceil(路线距离 × 路线耗时系数)
每帧移动量     = floor(基础每帧移动量 × 1000 ÷ 天气通行倍率)
```

| 路线类型 | 耗时系数 | 每帧鲜度损耗 |
|----------|----------|--------------|
| ROAD | 1380 | 0.055 |
| WATER | 1250 | 0.045 |
| MOUNTAIN | 1780 | 0.07 |
| BRANCH | 1550 | 0.065 |

| 加速 | 基础每帧移动量 | 持续帧 |
|------|----------------|--------|
| 无 | 1000 | - |
| 快马 FAST_HORSE | 1200 | 20 |
| 短程马 SHORT_HORSE | 1150 | 14 |
| 疾行令 RUSH_SPEED | 1300 | 15（与马互斥） |

| 天气 | 通行倍率 | 鲜度系数 |
|------|----------|----------|
| 酷暑 HOT（全图） | 1000 | ×1.5 |
| 暴雨 HEAVY_RAIN（水路） | 1350 | ×1.3 |
| 山雾 MOUNTAIN_FOG（山路） | 1100 | ×1 |

### 3.3 路径规划算法要求

1. **动态最短路**：每帧根据 `blockedNodes`（敌方设卡 + 道路障碍）重算
2. **天气感知**：收到 `weather.forecast` 后，对水路/山路边加权惩罚
3. **移动中状态**：`MOVING`/`WAITING` 时仅 `WAIT`、继续 `MOVE` 到当前目标、或使用马类/疾行令
4. **不可原路返回**：在路线边上不能 `MOVE` 回本段起点

### 3.4 阻塞处理优先级

| 阻挡 | 优先方案 | 次选 |
|------|----------|------|
| 道路障碍 | 顺路 T04 任务（+30 分且清障） | 主车队 CLEAR（6 帧，1 好果）或绕路 |
| 敌方设卡 | 绕路 | 攻坚 BREAK_GUARD / 强制通行 FORCED_PASS |
| 设卡 + 障碍并存 | 强制通行时间税取较高者 | - |

**强制通行注意：**

- 仅对相邻阻挡节点有效
- 有敌方设卡 → 触发 PASS 窗口（3 拍）
- 仅障碍 → 不触发窗口，时间税障碍固定 8 帧
- 同一节点完成后不可再次强制通行

## 4. 固定处理与交付流程

### 4.1 固定处理站点（须 `PROCESS` 或专用动作）

| 节点 | 类型 | 帧数 | 可窗口 |
|------|------|------|--------|
| S02 | TRANSFER 前段交接 | 4 | 是 |
| S04 | BOARD 登船 | 7 | 是 |
| S05 | WATER_TRANSFER 水路换运 | 6 | 是 |
| S11 | PASS_TRANSFER 入关交接 | 5 | 是 |
| S13 | PALACE_TRANSFER 宫前交接 | 5 | 是 |
| S14 | VERIFY 宫门验核 | 6 | 是（仅 RUSH 后可提交） |

- 未完成固定处理 **不可离站**
- 再次到达同一站需重新处理
- 探路标记：处理帧 −3，最低 2 帧

### 4.2 交付状态机（代码应实现为显式 FSM）

```text
[NORMAL] 移动/任务/资源 → 到达 S14 前完成 S11/S13 等处理
    ↓ phase=RUSH
[AT_S14] VERIFY_GATE（可窗口）→ verified=true
    ↓ MOVE
[AT_S15] DELIVER（需 goodFruit>0, freshness>0, 非移动/处理/窗口/休整中）
    ↓
[DELIVERED] 仅 WAIT 或 actions:[]
```

**S15 特殊规则：**

- 未验核或不满足交付：只能 WAIT 或返回 S14
- 返回 S14 可无视 S14 设卡与障碍

## 5. 皇榜任务策略

### 5.1 任务分值与性价比

| 模板 | 名称 | 分 | 处理帧 | 分/帧 | 备注 |
|------|------|-----|--------|-------|------|
| T01 | 限时过关 | 30 | 3 | 10.0 | 高优先，常 S03 |
| T06 | 争马换乘 | 30 | 3 | 10.0 | 需消耗 1 匹马 |
| T02/T08/T11 | - | 30 | 4 | 7.5 | 顺路做 |
| T04 | 清障 | 30 | 6 | 5.0 | 障碍在己方路径上必做 |
| T12/T13/T14 | - | 15 | 5 | 3.0 | 仅极度顺路 |

### 5.2 任务调度规则

1. **目标**：普通任务基础分 ≥ 90（3×30），有余力冲 110 拿满里程碑
2. **顺路原则**：偏离主路径 > N 帧（建议 15）的任务不做
3. **T04**：必须 `CLAIM_TASK` 才得分；`CLEAR`/小分队清障不得分
4. **T06**：无马不做
5. **过期**：关注 `expireRound`；窗口平局会暂停任务倒计时
6. **保护期**：`protectionPlayerId` 非己方时跳过
7. **同时仅处理 1 个任务实例**

## 6. 资源策略

### 6.1 资源效果

| 资源 | 用途 | 代码触发条件建议 |
|------|------|------------------|
| ICE_BOX | 鲜度 +10（先加后扣本帧） | 鲜度 < 72 或酷暑/山路前 |
| FAST_HORSE | 20 帧 ×1200 | 官道长途段起点 |
| SHORT_HORSE | 14 帧 ×1150 | 中短距离 |
| INTEL | 探路标记，距离 ≤15 | 下一处理点预标记 |
| PASS_TOKEN / OFFICIAL_PERMIT | 窗口验牒 YAN_DIE | 保留 1–2 给 GATE/RESOURCE |
| BOAT_RIGHT | 标记，不可主动使用 | 仅领取 |
| 护卫行动点（初始 4） | 兵争 BING_ZHENG | 高价值窗口 |

### 6.2 领取优先级（同站多资源）

`FAST_HORSE` > `SHORT_HORSE` > `ICE_BOX` > `OFFICIAL_PERMIT` > `PASS_TOKEN` > `INTEL` > `BOAT_RIGHT`

### 6.3 领取与窗口

- 领取读条 2 帧
- 同资源第 1 次被打断 → 资源窗口；第 2 次后手无效
- 胜方从头领取，不恢复旧进度

## 7. 窗口争夺策略

### 7.1 窗口类型

`RESOURCE` | `TASK` | `GATE` | `DOCK` | `PASS` | `OBSTACLE`

- 3 拍制：T 创建，T+1/2/3 出牌，T+3 结算
- 每队每帧最多 1 张窗口牌；多窗口时其他按弃权

### 7.2 克制关系

```text
验牒(YAN_DIE) 克 强行(QIANG_XING) 克 献贡(XIAN_GONG) 克 兵争(BING_ZHENG) 克 验牒
```

### 7.3 出牌决策表

| 窗口类型 | 建议 | 弃权条件 |
|----------|------|----------|
| GATE | 必争；有兵争/验牒/献贡 | 已验核 |
| RESOURCE（快马/官凭） | 争 | 非路线必需 |
| TASK（30 分） | 争 | 15 分且不顺路 |
| DOCK | 走水路必争 | 官道路线 |
| PASS | 攻方：强行/验牒；守方：按成本 | 时间税过高 |
| OBSTACLE | 仅 T04 在且值 30 分时争 | - |

**成本不足 → 牌失效 → 按弃权**

### 7.4 平局与冷却

- 非 PASS 窗口平局：双方休整 3 帧；任务倒计时暂停
- 同对象连续平局 2 次：冷却 18 帧（宫门 6 帧）
- 战术性平局：可拖对手 3 帧 + 冷却（慎用，勿耽误己方交付）

## 8. 对抗策略（设卡 / 攻坚 / 小分队）

### 8.1 设卡 SET_GUARD

- 每队最多 2 个有效设卡；完成下帧才生效
- 关键关隘（如 S10）：成本 1 好果，防守上限 7
- S14 宫门：成本 1 好果，上限 4（冲刺卡对手验核）
- **不要设在自己还要走的路线上**

### 8.2 攻坚 BREAK_GUARD

- 好果 2 点/篓，坏果 3 点/篓；单次各最多 2
- 成功：防守值清零；失败：休整 5 帧，防守值减少
- RUSH 后可绑 `BREAK_ORDER`（+3 攻坚值）

### 8.3 破关悬赏

- 普通 10 分 / 关键关隘 18 分；完成任意后额外 +20
- **仅当攻破时总分低于设卡方**才得悬赏分
- 领先时可“钓鱼”设卡；落后时集中破关

### 8.4 小分队（初始 8 人手，RUSH 后禁止新派）

| 动作 | 人手 | 场景 |
|------|------|------|
| SQUAD_SCOUT | 1 | 远程关键处理点探路 |
| SQUAD_CLEAR | 2 | 主车队不必到场清障（无 T04 分） |
| SQUAD_REINFORCE | 2 | 己方关键设卡 +2 防守 |
| SQUAD_WEAKEN | 2 | 敌方 S10/S11 设卡 −2（性价比高） |

## 9. 鲜度与果品管理

### 9.1 好果转坏阈值

鲜度**首次**低于 90/80/70/60/50/40/30/20/10 各转 1 篓坏果（等于不触发）。

### 9.2 保鲜策略

- 酷暑全图：冰鉴 + 尽量避山路
- 暴雨：避水路移动与登船/换运（处理 +4 帧）
- 护果令 RUSH_PROTECT：30 帧鲜度 ×0.2，停靠节点使用
- 疾行令与马互斥；疾行令鲜度 ×1.25

### 9.3 成本动作与好果预算

| 动作 | 好果成本 |
|------|----------|
| 主车队 CLEAR | 1 |
| 关键关隘设卡 | 1 |
| 疾行令 | 2 |
| 窗口献贡 XIAN_GONG | 1（且鲜度≥80） |

交付前保留：**好果 > 0 且鲜度 > 0**（否则无法 DELIVER）

## 12. 协议与工程约束（代码必须满足）

### 12.1 通信

- TCP：`5 位十进制长度 + UTF-8 JSON`
- 流程：`registration → start → ready → inquire/action 循环 → over`
- `action.round` **必须等于** `inquire.round`
- 无动作：`actions: []`（有效心跳）
- 连续 60 帧缺动作 → 退赛

### 12.2 每帧动作额度

| 类别 | 上限 |
|------|------|
| 主车队 | 1 |
| 小分队 | 1 |
| 窗口出牌 | 1 |
| 终局急策 | 1 |

### 12.3 状态 → 允许动作（简化）

| 状态 | 允许 |
|------|------|
| IDLE | 全部合法主车队动作 |
| MOVING/WAITING | WAIT, MOVE 到目标, 马类, 疾行令 |
| PROCESSING/VERIFYING/FORCED_PASSING | 等待（发 [] 或 WAIT） |
| CONTESTING | WINDOW_CARD |
| RESTING | WAIT 或 [] |
| DELIVERED | WAIT 或 DELIVER |

### 12.4 提交包

- ZIP 根目录 `start.sh`，参数 `<playerId> <host> <port>`
- 不写死身份与地址
- 依赖随包，禁止运行时 install

## 14. 单帧决策伪代码

```text
onInquire(data):
  updateContext(data)
  if self.delivered: return [WAIT] or []

  if inActiveWindow(): return [WINDOW_CARD best or ABSTAIN]

  mode = OpponentModel.classify()

  if state in {PROCESSING, VERIFYING, FORCED_PASSING, RESTING, CONTESTING}:
    return []  # 心跳

  if state in {MOVING, WAITING}:
    if shouldUseHorse(): return [USE_RESOURCE horse]
    return []

  # IDLE at node
  if phase==RUSH and at S14 and not verified:
    return [VERIFY_GATE + optional BREAK_ORDER]

  if at S15 and verified and canDeliver():
    return [DELIVER]

  if needProcessAtCurrentNode():
    return [PROCESS or DOCK]

  if phase==RUSH and shouldUseRushTactic():
    return [RUSH_SPEED or RUSH_PROTECT]

  if shouldUseIceBox(): return [USE_RESOURCE ICE_BOX]

  task = TaskScheduler.pick(mode)
  if task: return [CLAIM_TASK task]

  res = ResourceManager.pickClaim()
  if res: return [CLAIM_RESOURCE res]

  if CombatAdvisor.shouldSetGuard(mode): return [SET_GUARD ...]
  if blocked(nextHop): return CombatAdvisor.breakOrPassOrReroute()

  next = RoutePlanner.nextHop(goalNode)
  if next: return [MOVE next]

  squad = SquadAdvisor.pick()  # if not RUSH
  return [] with optional squad
```
